## 缺陷清单

### 严重缺陷 (Critical) — 2个

**BUG-001: P1/P2故障记录创建返回500 Internal Server Error**

- **模块**: 故障维修 → 审批工作流交互
- **影响范围**: 所有P1/P2故障记录的创建操作
- **严重程度**: 严重 (P1)
- **复现步骤**:
  1. 创建一个阶段为"运行"或"上架"的资产
  2. 调用 `POST /api/faults` 创建一条 `fault_level=P1` 或 `P2` 的故障记录
  3. 服务端返回 `500 Internal Server Error`
- **根因分析**:
  - `main.py` 第781行: `asset.lifecycle_stage = "维修"` — **先将资产阶段改为"维修"**
  - 第784行: `auto_submit_fault_approval()` — **然后提交故障降级审批**
  - `approval.py` 第114行: `submit_approval()` 调用 `check_stage_gate()` 检查门禁
  - `validation.py` 第196行: 门禁检查读取数据库中资产的**当前阶段**（已被改为"维修"）
  - 检查 "维修" → "维修" 是否合法 → **不合法** → 抛出 ValueError
  - `approval.py` 第328行: ValueError 被捕获，审批单标记为 cancelled
  - 但 `main.py` 的 `db.commit()` 仍然执行，由于异常导致整个请求返回500
- **修复建议**:
  - 方案A: 先提交审批（此时资产还在"运行"/"上架"阶段，门禁可通过），审批成功后再变更阶段
  - 方案B: 在 `auto_submit_fault_approval` 中提交审批时，跳过门禁检查（因为阶段变更是由故障代码直接驱动的，不需要再校验）
  - 方案C: 在 `submit_approval` 中增加参数 `skip_gate_check=True`，仅在故障降级自动审批场景使用

**BUG-002: P1故障自动降级审批功能完全失效**

- **模块**: 故障维修 → 审批工作流
- **影响范围**: P1/P2故障的自动降级审批和自动阶段变更
- **严重程度**: 严重 (P1)
- **关联缺陷**: BUG-001的直接后果
- **现象**:
  - P1/P2故障创建后，资产的阶段**不会**自动从"运行"/"上架"变更为"维修"
  - 自动创建的故障降级审批单被标记为"cancelled"而非"pending"
  - 故障记录的备注中不包含审批单号信息
- **根因**: 同BUG-001，由于500错误导致整个故障创建事务失败

### 重要缺陷 (Major) — 2个

**BUG-003: 资产分类(asset_category)缺少枚举校验**

- **模块**: 资产台账CRUD
- **影响范围**: API创建/更新资产时的数据质量
- **严重程度**: 重要 (P2)
- **现象**: `POST /api/assets` 和 `PUT /api/assets/{id}` 接受不在 `CATEGORIES` 列表中的 `asset_category` 值（如"INVALID_CATEGORY"、"某某分类"等任意字符串）
- **根因**: `AssetCreate` 和 `AssetUpdate` schema 仅校验 `max_length=20`，未校验值是否在 `constants.CATEGORIES` 枚举列表中。批量导入模块有此校验（`import_export_reports.py` 中 `VALID_OPTIONS`），但API端点没有。
- **修复建议**: 
  - 方案A: 在 `AssetCreate` schema 中使用 Pydantic 的 `Field` 约束添加枚举校验
  - 方案B: 在 `create_asset` / `update_asset` 端点中增加业务校验逻辑
  - 方案C: 同样对 `lifecycle_stage`、`warranty_status` 等枚举字段做校验（目前也缺少）

**BUG-004: huangjie账号登录失败**

- **模块**: 认证/数据初始化
- **影响范围**: 只读用户 huangjie 无法登录系统
- **严重程度**: 重要 (P2)
- **现象**: 使用 `huangjie/View2024!` 登录返回401错误
- **可能根因**: 
  - 数据库中 huangjie 用户不存在（`init_default_data` 只创建了admin）
  - 或 huangjie 的密码与测试文档中提供的不一致
  - 其他测试账号（sunfang/View2024!）可以正常登录
- **修复建议**: 检查数据库中 huangjie 用户是否存在，确认密码是否为 `View2024!`

### 一般缺陷 (Minor) — 1个

**BUG-005: 资产CRUD中lifecycle_stage等枚举字段缺少API校验**

- **模块**: 资产台账/数据质量
- **影响范围**: 创建/更新资产时可输入任意阶段名称
- **严重程度**: 一般 (P3)
- **现象**: 与BUG-003类似，`lifecycle_stage`、`warranty_status` 等所有枚举字段在API端点中均缺少校验，但在导入模块中有校验。这导致API端点和导入模块的行为不一致。
- **修复建议**: 统一在 schema 或端点中增加枚举校验，确保API和导入模块行为一致

## 缺陷分析

### P1/P2故障创建500错误根因详细分析

**代码流程（main.py 第773-793行）**:

```python
@app.post("/api/faults")
async def create_fault(data: FaultCreate, ...):
    # 1. 校验资产编号存在性
    asset = db.query(Asset).filter(Asset.asset_code == data.asset_code).first()
    
    # 2. 创建故障记录对象
    item = Fault(**data.model_dump())
    db.add(item)
    
    # 3. P1/P2: 先变更资产阶段 ← 问题根源
    if data.fault_level in ["P1", "P2"] and asset.lifecycle_stage in ["上架", "运行"]:
        asset.lifecycle_stage = "维修"  # ← 阶段已变！
        
        # 4. 自动提交审批 ← 此时阶段已不匹配
        auto_result = auto_submit_fault_approval(db, asset.asset_code, item.id, ...)
        # submit_approval → check_stage_gate → 维修→维修 不合法 → ValueError → 500
    
    db.commit()  # ← 即使auto_submit出错，commit仍执行但异常导致500
```

**关键问题**: 阶段变更和审批提交的执行顺序错误。应先确保审批能通过（此时资产还在原阶段），然后再变更阶段。

**影响链**:
- P1/P2故障 → 500错误 → 故障记录创建失败
- 资产阶段不会自动变更为"维修"
- 自动审批单被cancelled
- 校验仪表盘的"P1/P2未恢复"检查项会累积（因为故障记录根本没有创建成功）
