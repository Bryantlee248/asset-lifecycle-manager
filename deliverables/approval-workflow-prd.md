## 产品定义

### Product Goals

1. **合规达标**：满足等保2.0三级要求"变更需经审批后实施"，系统可自动生成审批证据链（提交人→审批人→审批意见→审批时间→阶段变更记录），消除当前合规风险
2. **流程驱动**：资产阶段变更由审批结果自动触发，取代当前人工直接修改lifecycle_stage字段的方式，确保"未经审批不得变更阶段"
3. **可追溯性**：每一笔审批操作留痕（谁审批了什么、何时、结果如何），与现有审计日志体系集成，形成完整变更证据链

### User Stories

|---|---------|---------|
| US1 | 作为运维经理，我想要提交采购立项审批单并经上级审批通过后，系统自动将资产阶段从"规划"流转到"在途"，以便确保采购行为合规可追溯 | 规划→在途（采购立项审批） |
| US2 | 作为运维工程师，我想要提交验收确认审批单，审批通过后系统自动将资产阶段从"在途"流转到"上架"，以便确保设备上线经过正式验收 | 在途→上架（验收确认审批） |
| US3 | 作为运维工程师，我想要提交故障降级审批单（P1/P2故障自动触发），审批通过后系统自动将资产阶段从"运行"流转到"维修"，以便快速响应严重故障 | 运行→维修（故障降级审批） |

## 技术规范

### Requirements Pool

#### P0 — Must Have（核心流程引擎）

| ID | 需求 | 描述 |
|----|------|------|
| P0-01 | 审批单模型（ApprovalRequest） | 独立审批实体，包含：申请类型（6种变更流程对应）、申请人、申请时间、目标资产ID、原阶段→目标阶段、审批状态（draft/pending/approved/rejected/cancelled）、审批意见、审批时间 |
| P0-02 | 状态机引擎 | 审批单生命周期状态流转：draft → pending → approved/rejected，rejected可重新提交（draft），cancelled为主动撤回 |
| P0-03 | 审批通过驱动阶段变更 | 审批单approved后，自动调用现有阶段门禁（check_stage_gate）校验，校验通过后更新assets.lifecycle_stage字段并写入审计日志 |
| P0-04 | 与阶段门禁集成 | 审批提交时自动校验目标阶段是否合法（调用check_stage_gate），非法跳转在提交阶段即被拦截 |
| P0-05 | 审批权限集成 | 审批提交需asset_change_submit权限，审批操作需asset_change_approve权限，与现有RBAC体系无缝衔接 |
| P0-06 | 6种审批类型枚举 | procurement_approval / inspection_approval / fault_degrade_approval / migration_approval / warranty_renewal_approval / retirement_approval |

#### P1 — Should Have（审批链与通知）

| ID | 需求 | 描述 |
|----|------|------|
| P1-01 | 多级审批链配置 | 报废退役审批支持逐级审批链（运维工程师→运维经理→主管），每级审批通过后流转至下一级，任一级驳回即终止 |
| P1-02 | 审批人自动指派 | 根据审批类型+申请人角色自动确定审批链：单级审批直接指派上级角色，多级审批按预设链路逐级指派 |
| P1-03 | 站内审批通知 | 待审批单创建后向审批人推送站内通知（Notification模型），显示在导航栏"待办"徽标中 |
| P1-04 | 审批结果通知 | 审批通过/驳回后向申请人推送站内通知，告知审批结果与审批意见 |
| P1-05 | P1/P2故障自动提交 | fault_level为P1或P2时，系统自动创建故障降级审批单并进入pending状态，减少严重故障响应延迟 |

#### P2 — Nice to Have（追溯与报表）

| ID | 需求 | 描述 |
|----|------|------|
| P2-01 | 审批历史追溯 | 每笔审批单可查看完整审批轨迹（提交→每级审批→结果），支持按资产/时间/类型筛选 |
| P2-02 | 审批统计报表 | 统计审批通过率、平均审批耗时、各类型审批量分布，为管理决策提供数据支撑 |
| P2-03 | 审批单导出 | 支持审批记录导出为PDF/Excel，满足等保审计检查需要提供书面证据的场景 |

### UI Design Draft

#### 核心交互流程

```
提交审批 → 审批人审批 → 审批结果 → 阶段自动变更
```

#### Mermaid 流程图

```mermaid
flowchart TD
    A[申请人点击"提交审批"] --> B{选择审批类型}
    B --> C[填写审批申请表单]
    C --> D{阶段门禁校验}
    D -- 校验失败 --> E[提示不满足前置条件,返回修改]
    D -- 校验通过 --> F[创建审批单 pending状态]
    F --> G[通知审批人]
    G --> H[审批人进入"我的待审"列表]
    H --> I[审批人查看审批详情]
    I --> J{审批决策}
    J -- 驳回 --> K[审批单状态 rejected]
    K --> L[通知申请人 驳回原因]
    L --> M[申请人可修改后重新提交]
    J -- 通过 --> N{是否多级审批?}
    N -- 是,仍有下一级 --> O[流转至下一级审批人]
    O --> G
    N -- 否,全部通过 --> P[审批单状态 approved]
    P --> Q[调用阶段门禁二次校验]
    Q -- 校验通过 --> R[自动更新 assets.lifecycle_stage]
    R --> S[写入审计日志]
    S --> T[通知申请人 审批通过]
    Q -- 校验失败 --> U[审批通过但阶段变更异常,需人工介入]
```

#### 页面设计描述

1. **审批提交页面**（资产详情页内嵌入口）
   - 资产详情页增加"发起阶段变更审批"按钮，点击弹出审批类型选择对话框
   - 根据当前资产阶段自动筛选可用的审批类型（如运行阶段资产可发起故障降级/变更迁移/报废退役审批）
   - 表单字段：审批类型（下拉选择）、变更原因（文本域）、附件上传（支持截图/文档）
   - 提交前自动调用阶段门禁校验，校验失败在表单内inline提示具体不满足的条件

2. **我的待审列表页**（新增独立页面）
   - 顶部筛选栏：审批类型、资产编号、申请时间范围
   - 列表字段：审批单号、审批类型、资产编号/名称、申请人、申请时间、当前审批级别、紧急程度
   - 未读待审高亮显示，支持批量审批（仅限同一类型且同级别）
   - 点击行进入审批详情页

3. **审批详情页**（新增独立页面）
   - 上半部分：审批申请信息（申请人、资产信息、变更原因、附件）
   - 中间部分：审批链路可视化（显示当前处于第几级审批、每级审批人及状态）
   - 下半部分：审批操作区（通过/驳回按钮 + 审批意见文本域）
   - 驳回时必须填写驳回原因

4. **审批历史页**（资产详情页内嵌入口）
   - 资产详情页增加"审批记录"Tab，展示该资产所有审批单时间线
   - 每条记录显示：审批类型、申请时间、审批结果、每级审批人+意见+时间
   - 支持展开查看完整审批链路

5. **导航栏待办徽标**
   - 顶部导航栏增加"待办审批"图标+未读数量徽标（类似邮件未读数）
   - 点击跳转至我的待审列表页

### Open Questions

| # | 问题 | 影响范围 | 建议 |
|---|------|---------|------|
| Q1 | 报废退役审批的多级审批链具体角色顺序是什么？ | P1-01 多级审批配置 | 建议默认链路：运维工程师(提交) → 运维经理(一级审批) → 主管(二级审批)，可配置 |
| Q2 | 审批驳回后是否允许无限次重新提交，还是设置驳回次数上限？ | P0-02 状态机设计 | 建议允许重新提交但记录驳回次数，超过3次驳回自动标记为"争议单"需主管介入 |
| Q3 | P1/P2故障自动创建审批单后，是否需要人工确认才进入pending，还是直接进入pending？ | P1-05 自动提交 | 建议直接进入pending以加速严重故障响应，但审批人可在审批时驳回 |
| Q4 | 审批人外出/请假时的审批代办机制是否纳入本期？ | P1 审批链 | 建议本期暂不实现代办转签，作为P1.5在下期迭代 |
| Q5 | 审批通知是否需要支持邮件/短信渠道，还是仅站内通知？ | P1-03/04 通知范围 | 建议本期仅站内通知，邮件/短信通道作为P2延伸 |
| Q6 | 审批单号编号规则：是否与现有application_no字段统一？ | P0-01 模型设计 | 建议新审批单号采用APR-YYYYMMDD-SEQ格式，与退役表application_no兼容但区分命名空间 |
| Q7 | 变更迁移审批通过后，是否需要在changes表同步记录，还是仅依赖审批单+审计日志？ | P0-03 阶段变更联动 | 建议审批通过后同步更新changes表completion_status，保持现有数据模型一致性 |

## 数据模型草案（供架构师参考）

```python
class ApprovalRequest(BaseModel):
    id: int                          # 审批单ID
    request_no: str                  # 审批单号 APR-YYYYMMDD-SEQ
    approval_type: str               # 审批类型枚举（6种）
    asset_id: int                    # 目标资产ID
    current_stage: str               # 当前阶段
    target_stage: str                # 目标阶段
    reason: str                      # 变更原因
    attachments: str                 # 附件路径(JSON)
    status: str                      # draft/pending/approved/rejected/cancelled
    applicant_id: int                # 申请人ID
    applied_at: datetime             # 申请时间
    current_level: int               # 当前审批级别(多级审批)
    rejection_count: int             # 驳回次数

class ApprovalStep(BaseModel):
    id: int                          # 步骤ID
    request_id: int                  # 关联审批单ID
    level: int                       # 审批级别(1=N级)
    approver_id: int                 # 审批人ID
    approver_role: str               # 审批人角色
    status: str                      # pending/approved/rejected
    comment: str                     # 审批意见
    acted_at: datetime               # 审批操作时间

class ApprovalNotification(BaseModel):
    id: int                          # 通知ID
    request_id: int                  # 关联审批单ID
    user_id: int                     # 接收人ID
    type: str                        # pending_approval/approved/rejected
    title: str                       # 通知标题
    content: str                     # 通知内容
    is_read: bool                    # 是否已读
    created_at: datetime             # 创建时间
```
