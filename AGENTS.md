# IT 资产全生命周期管理系统开发规则

本文件适用于在本仓库工作的所有 AI 和开发者。开始任何任务前必须完整阅读，并以本文件为准执行。

## 1. 工作边界

- 当前项目根目录为 `asset-lifecycle-manager`，先执行 `git status --short`、`git log -1 --oneline` 和阅读 `README.md`，确认工作基线。
- 未经用户明确授权，不执行 `git commit`、`git push`、部署、重启服务、修改远端主机、访问公网地址或更改生产数据库。
- 未经明确授权，不执行 `git reset --hard`、`git clean`、`git checkout --`、`git worktree remove --force`、递归删除或覆盖 `.env`、数据库、密钥和备份文件。
- 发现非本人产生的修改、未跟踪文件或运行中的服务时，保留它们并报告；不得擅自清理或还原。

### 1.1 多 Agent 协作与当前恢复基线

- 唯一允许直接开发的项目根目录为 `D:\Codex Project\Report-Modified\asset-lifecycle-manager`。不得在其父目录、旧工作树或 `deliverables/software-company/` 中修改运行代码。
- 当前工作区包含尚未提交的恢复改动。任何 Agent 开始工作前必须执行 `git status --short` 并保留现有修改；不得覆盖、删除、暂存或提交非本次任务产生的文件。
- 未经用户当次明确授权，禁止执行 `git reset --hard`、`git clean`、`git checkout --`、`git restore`、强制删除工作树、递归删除文件，或用外部副本覆盖本目录。
- 未经用户当次明确授权，禁止 `git commit`、`git push`、创建或合并分支、部署、重启服务、访问公网地址、访问远端主机，或修改生产数据库与配置。
- 不得把 `127.0.0.1:8000` 作为 QA、smoke 或数据生成脚本的默认目标。任何会写入或删除数据的脚本必须遵循第 5 节的显式本地目标和 `--destructive` 要求。
- 完成开发后必须报告本次实际修改的文件、验证命令及结果、已知限制，以及是否执行了提交、推送或部署。默认保持改动未提交、未推送、未部署。

### 1.2 外部 AI 开发门禁

- 多页面改造、跨模块功能和上线前修复必须分为两阶段：先仅输出详细实施方案，方案经审核确认后才能改动代码；方案必须列出修改文件、保留行为、测试策略、验收规则、风险和回滚方式。
- 通过方案审核后，按独立批次实施和验收。每个批次必须有明确输入、输出、自动化测试、浏览器或功能验证证据；不得用一次全系统“已完成”声明替代分批验证。
- 外部 AI 无法验证、缺少运行条件或发现需求冲突时，必须停止并报告阻塞原因，不得猜测、跳过验证或用局部改动宣称完成。
- 评审人员应以实际差异、测试输出和可复核功能证据决定是否通过；实施方案不能替代实现后的验证。

## 2. 开发前要求

- 先明确目标、影响范围、验收标准和是否涉及数据迁移、权限、配置、部署或前端交互。存在歧义时先提问，不自行扩大范围。
- 优先复用既有 FastAPI、SQLAlchemy、Vue、Element Plus、配置缓存和测试模式；不为一次性需求引入新框架或抽象层。
- 只修改完成需求所必需的文件。禁止顺带重排无关格式、重写大文件或删除“看起来无用”的历史文件。
- 所有新增行为先写最小自动化测试并确认其在实现前失败；实现后运行对应测试和完整回归。

## 3. 后端、数据库与配置

- `backend/database.py` 的 ORM 字段、`backend/schemas.py`、路由实现、导入导出逻辑、测试夹具和测试数据生成器必须同步更新。删除或替换字段前，先全仓搜索旧字段引用。
- 修改 SQLite 连接参数时，必须验证外键、超时、日志模式和并发行为；不能只修改实现而不更新对应测试。
- 新增表、列、索引或约束时，提供可重复执行、保留存量数据的迁移或 seed，并说明回滚方式。不得通过删库重建来完成升级。
- 配置中心写操作必须使用 `config:manage` 权限，并在提交后调用 `invalidate_and_rebuild`。系统 seed 必须幂等，默认配置必须保持既有行为。
- 修改阶段流转、校验规则、聚合白名单或审批流时，必须覆盖允许路径、拒绝路径、权限路径和缓存失效路径。

## 4. 前端规则

- 前端为单文件 Vue 页面 `frontend/index.html`。保留既有 API 路径、权限判断、`currentTab` 值、表单字段、CRUD 行为和现有页面入口，除非需求明确要求变更。
- 页面改造必须同时覆盖桌面和窄屏断点，避免遮挡、横向溢出和动态内容导致的布局抖动。
- 使用 Element Plus 与已引入的图标库；禁止新增默认密码、令牌、接口地址或环境差异到前端源码。
- 修改公共视觉令牌、导航或脚本加载顺序时，更新相应静态契约测试并进行浏览器级验证。

## 5. 测试、数据与 QA

- 默认验收命令为：

  ```powershell
  python -m pytest -q --disable-warnings
  python -m compileall backend -q
  git diff --check
  ```

- 默认测试必须在隔离数据库或显式指定的临时端口中运行。不得把 `127.0.0.1:8000`、管理员账号或密码写死为默认目标。
- 任何会创建、更新、删除数据的 QA 或 smoke 脚本必须同时满足：显式 `--base-url`、默认拒绝非本地目标、`--destructive` 二次确认、独立测试前缀、完成后的可验证清理或备份恢复。
- 场景测试数据必须使用独立前缀（当前为 `DC-CL-SIM-`），追加且幂等，不得清空既有资产或业务记录。
- 测试失败时先定位根因。不得通过缩小测试收集范围、删除失败断言或跳过测试来制造“通过”。

## 6. 安全、凭据与部署

- 禁止将真实或示例可登录密码、JWT、SSH 私钥、Token、数据库文件、备份或生产地址提交到 Git、README、QA 脚本、前端或 CI 日志。
- 机密只可通过 `.env`、GitHub Secrets 或部署平台密钥注入；文档仅使用占位符，例如 `replace-with-a-strong-password`。
- CI 必须至少执行依赖安装、编译、完整 `pytest` 和生产配置导入检查。部署任务只能在 CI 成功后执行。
- 数据库迁移、备份恢复、systemd 和 Nginx 改动必须先在本地或隔离环境验证。涉及生产的操作必须获得用户当次明确确认。

## 7. 交付与沟通

- 交付前报告：修改文件、行为变化、测试命令与结果、已知限制，以及是否执行了提交、推送或部署。
- 没有新鲜验证证据时，不得声称“已完成”“可上线”或“测试通过”。
- 除非用户明确要求，保持改动未提交、未推送、未部署。
